# 1.4-alpha
指完成了第四节要求的版本。先前的版本管理较为混乱。

## 本版已实现功能

- 基于服务端转发的 1 对 1 通信结构
- 图形化客户端主界面
- 在线用户列表显示
- 目标用户选择
- 文本消息发送
- 文件发送
- 离线语音录制与发送
- 实时语音通话
- 呼叫 / 接听 / 挂断 的基础通话状态机
- 联系人保存、备注、删除
- 服务端基础管理命令
- 项目结构已整理为 `src/` + `data/` 形式

## 当前目录说明

- `src/apps/`：客户端界面入口
- `src/server/`：服务端入口
- `src/core/`：协议、配置、音频、联系人等核心模块
- `data/contacts/`：联系人数据
- `data/received/`：客户端接收的文件/音频
- `data/recorded/`：本地录音
- `data/server_received/`：服务端收到的音频

## 运行方式

### 1. 启动服务端

直接双击根目录下：

- `start_server.bat`

或在终端运行：

```bash
python -m src.server.server
```

### 2. 启动客户端

直接双击根目录下：

- `start_client.bat`

或在终端运行：

```bash
python -m src.apps.client_stream_gui
```

## 基本操作指南

### 客户端

1. 启动客户端后，先输入用户名。
2. 连接成功后，在左侧列表中选择一个在线用户作为当前目标。
3. 可直接进行以下操作：
   - 发送文字
   - 发送文件
   - 录音 5 秒发送
   - 保存联系人 / 添加备注 / 删除联系人
4. 若要进行实时语音通话：
   - 先选择目标用户
   - 点击“呼叫”
   - 对方点击“接听”
   - 接通后再点击“开启实时通话”
   - 通话结束后点击“挂断”

### 服务端

服务端启动后，可在控制台使用以下命令：

- `help`：查看命令
- `list`：查看在线客户端
- `all <消息>`：向所有客户端广播文本
- `to <用户名> <消息>`：向指定用户发送文本
- `sendaudio <用户名> <路径>`：向指定用户发送音频文件
- `playlast`：播放最近收到的音频
- `playaudio <路径>`：播放指定音频文件
- `quit`：关闭服务器

**潜在问题**

用户名目前会被“后登录者直接顶掉”，没有真正注册校验，这是当前最明显的功能风险。`server.py` 中如果新连接用了一个已存在的用户名，服务端会直接关闭旧连接并把这个名字给新连接。这会导致“同名登录踢人”、状态混乱，也和 README 里“加用户注册”的方向不一致。

来电状态机会被第三方来电直接覆盖，正在通话或正在呼叫时没有“忙线/拒绝新来电”处理。收到 `call` 后会直接把本地状态切成 `RINGING`。如果用户此时已经在 `CALLING` 或 `TALKING`，当前会话状态会被覆盖，实时语音按钮和通话对象也可能随之错乱。

`hangup` 现在可以对“当前选中的任意用户”发送，而不一定是实际通话对象。也就是说只要选中了某个人，即使没有建立通话，也能给对方发 `hangup`，而对方收到后会直接重置通话状态。

联系人目录没有在保存前保证存在，干净环境下第一次保存联系人可能失败。当前联系人保存逻辑直接写 `CONTACTS_FILE`，但没有在写入前显式确保父目录存在。你现在本地已有 `data/contacts/` 所以不明显，但新同学第一次运行时可能会报错。

文件和音频发送目前都是整文件读入内存，功能能跑，但扩展性偏弱。现在是一次性 `read()` 后直接发送；以后如果发送较大文件，会卡 UI、占内存，也不利于做进度提示。
